## 1. 用户角色定义

### 1.1 实际审核人员

实际审核人员是审核工作的 **核心执行者**，包含审核工程师与现场审核人员等一线专业人员。

他们通常具备扎实的行业知识、熟悉审核标准与业务规则，日常工作围绕「发现问题 → 分析判断 → 输出结论 → 推动整改」展开。他们既要保证审核结论的专业性与准确性，又要在有限时间内处理大量审核任务。

在本系统中，实际审核人员是 **主要使用者**，承担审核任务从创建到闭环的全过程推进。

| 维度 | 描述 |
|------|------|
| 工作性质 | 专业分析型，强调判断与结论 |
| 使用频率 | 高频，每日多次使用 |
| 关注重点 | 审核效率、问题准确性、报告质量、整改跟进 |
| 决策权限 | 对审核结论、问题认定、报告内容拥有确认权 |

### 1.2 审核管理人员

审核管理人员是审核工作的 **监督者与决策者**，通常担任部门负责人、质量总监或业务线管理者，并拥有对组织内审核数据的全量查阅权限。

他们不直接参与每一条审核细节，但需要掌握整体审核态势，关注高风险问题与整改进展，并据此做出管理决策或资源调配。

在本系统中，审核管理人员与实际审核人员使用 **同一套审核项目 / 复审 / 报告流程**，差异主要在 **数据权限范围**。管理所需的负荷、状态分布、工厂与问题趋势等，在 **审核项目列表** 的统计区查看，**不设独立驾驶舱**。

| 维度 | 描述 |
|------|------|
| 工作性质 | 管理决策型，强调全局视角与风险把控 |
| 使用频率 | 中低频，定期查阅或遇重大事项时查看 |
| 关注重点 | 列表统计、复审结论、报告中心四类产出 |
| 决策权限 | 对重大风险问题的处置方向、整改优先级拥有决策权 |
| 与执行者差异 | 同界面；通常可见更多工厂/项目数据 |

## 2. 用户工作场景

### 2.1 实际审核人员的典型工作场景

#### 场景一：接到新的审核任务

实际审核人员在周一早晨收到通知：某供应商的季度质量审核需要在本周内完成。他需要了解审核对象、审核范围、适用标准，并开始准备审核材料。

**现实状态：** 他需要翻阅邮件、查找历史审核记录、确认本次应适用的审核条款，往往耗费大半天时间才能进入实质性分析。

#### 场景二：整理与分析审核材料

工程师手上有供应商提交的质量报告、检测数据、流程文件、历史整改记录等大量材料。他需要逐份阅读，对照审核标准逐项核对，标记可疑之处。

**现实状态：** 材料分散在邮件附件、共享文件夹、纸质文档中，核对过程枯燥且容易疲劳，复杂条款的比对尤其耗时。

#### 场景三：识别问题并判断风险

在分析过程中，工程师发现若干不符合项或潜在风险点。他需要判断每个问题的严重程度、是否属于重复发生、应引用哪条标准作为依据。

**现实状态：** 判断高度依赖个人经验，新人难以独立处理；相似问题在不同审核中的认定标准可能不一致。

#### 场景四：撰写审核报告

分析完成后，工程师需要将发现的问题、证据材料、风险判断、整改建议等内容整理成正式的审核报告，提交给相关方。

**现实状态：** 报告编写是重复性最高的环节，大量时间花在格式整理、内容汇总、证据引用上，而非专业判断本身。

#### 场景五：跟踪整改进度

审核报告发出后，工程师需要跟进被审核方的整改情况，确认整改措施是否到位，必要时进行复核。

**现实状态：** 整改进度往往通过邮件、电话、表格等方式跟踪，信息分散，容易遗漏或延误。

### 2.2 审核管理人员的典型工作场景

#### 场景一：定期审阅审核成效

每月或每季度，管理人员需要了解本部门或业务线的审核完成情况、问题分布、整改闭环率等整体数据。

#### 场景二：关注高风险问题

当审核中发现重大或高风险问题时，管理人员需要第一时间获知详情，评估影响范围，并推动快速处置。

#### 场景三：评估整改效果

管理人员需要确认历史问题的整改是否真正到位，是否存在「屡改屡犯」的情况，以判断质量管理措施的有效性。

## 3. 用户核心需求

### 3.1 实际审核人员的核心需求

| 需求 | 说明 |
|------|------|
| **快速进入审核状态** | 接到任务后，能迅速明确审核对象、范围与适用标准，无需大量前期准备 |
| **高效处理审核材料** | 系统帮助整理、解析材料，自动提取关键信息，减少逐份翻阅的时间 |
| **可靠的问题识别辅助** | AI 帮助发现潜在问题并给出依据，工程师专注于确认与判断，而非从零搜寻 |
| **标准化的风险判断** | 问题风险等级有清晰参考，减少因个人经验差异导致的判断偏差 |
| **快速生成规范报告** | 基于分析结果自动生成报告初稿，工程师只需审核、修订与确认 |
| **便捷的整改跟进** | 整改任务、进度、复核结果在同一平台内闭环管理，无需跨工具追踪 |

### 3.2 审核管理人员的核心需求

| 需求 | 说明 |
|------|------|
| **一目了然的审核概览** | 无需深入细节，即可掌握审核整体进展与结果分布 |
| **高风险问题及时触达** | 重大风险问题主动呈现，无需等待工程师逐级汇报 |
| **整改闭环可追踪** | 清楚看到哪些问题已整改、哪些仍在进行中、哪些存在反复 |
| **数据支撑管理决策** | 审核数据以直观方式呈现，辅助资源配置与质量策略调整 |

## 4. 用户痛点

### 4.1 实际审核人员的痛点

| 痛点 | 业务表现 | 影响 |
|------|----------|------|
| **规则查找困难** | 审核标准散落在各类文档中，每次审核都要重新确认适用条款 | 前期准备时间长，执行标准不一致 |
| **材料分析负担重** | 大量文档和数据需人工逐份阅读、比对、摘录 | 有效分析时间被挤压，容易疲劳出错 |
| **问题发现靠经验** | 复杂或隐蔽问题高度依赖个人积累，新人难以独立完成 | 审核质量因人而异，存在遗漏风险 |
| **报告编写占时过多** | 大量精力花在格式整理、内容汇总、证据编排上 | 专业判断时间被挤占，报告交付延迟 |
| **整改跟踪分散** | 整改进度通过邮件、表格、口头沟通等方式跟进 | 信息易丢失，闭环周期长，反复问题难以及时发现 |

### 4.2 审核管理人员的痛点

| 痛点 | 业务表现 | 影响 |
|------|----------|------|
| **信息获取滞后** | 依赖工程师逐级汇报，难以实时掌握审核动态 | 高风险问题响应不及时 |
| **缺乏全局视角** | 审核数据分散在各处，难以汇总分析 | 管理决策缺乏数据支撑 |
| **整改效果难评估** | 不清楚历史问题的整改是否真正到位 | 重复问题反复发生，管理措施难以见效 |

## 5. 用户使用流程

### 5.1 实际审核人员完成一次审核任务的全流程

以下以实际审核人员 **完成一次供应商季度质量审核** 为例，描述其真实业务使用流程。

---

#### 第一步：创建审核任务

工程师登录系统，新建一条审核任务。他填写审核对象（某供应商）、审核类型（季度质量审核）、审核时间范围等基本信息。

系统根据审核类型，自动推荐本次应适用的审核标准与规则条款，工程师确认或微调后即可开始。

> **业务价值：** 无需翻阅大量文档确认适用标准，快速进入工作状态。

---

#### 第二步：上传审核材料

工程师将供应商提交的质量报告、检测数据、流程文件等材料上传至任务中。材料可能来自邮件附件、共享盘或纸质扫描件。

系统对材料进行智能解析，自动提取关键信息，并按审核维度进行分类整理。

> **业务价值：** 材料集中管理，关键信息自动呈现，无需逐份打开阅读。

---

#### 第三步：AI 辅助分析

系统基于已匹配的审核规则与知识库，对材料进行智能分析，自动识别潜在问题，并标注对应的规则依据与证据来源。

工程师进入分析界面，查看 AI 识别的问题清单。他可以逐条审阅：确认属实的问题予以采纳，误报的问题予以驳回，遗漏的问题手动补充。

> **业务价值：** 工程师从「从零搜寻问题」转变为「确认与判断问题」，大幅提升分析效率与覆盖面。

---

#### 第四步：风险判断与问题确认

对于已确认的问题，系统给出初步风险等级建议。工程师结合业务实际情况，调整风险等级，补充专业判断说明。

全部问题确认完成后，形成本次审核的问题清单，作为报告编写的基础。

> **业务价值：** 风险判断有统一参考标准，减少主观偏差，问题清单结构化沉淀。

---

#### 第五步：生成审核报告

工程师触发报告生成，系统基于问题清单、分析结论、证据材料，自动生成结构化的审核报告初稿。

工程师审阅报告内容，对表述、结论、建议进行修订与确认，定稿后提交或导出。

> **业务价值：** 报告编写从数小时缩短至数十分钟，工程师专注于结论质量而非格式整理。

---

#### 第六步：发起整改跟踪

对于需要整改的问题，工程师在系统中创建整改任务，指定责任方与完成时限。

整改过程中，工程师可随时查看进度，接收整改完成通知，并进行复核确认。

> **业务价值：** 整改全流程在同一平台闭环，无需跨工具追踪，问题处置有据可查。

---

#### 第七步：任务归档

整改复核通过后，工程师将本次审核任务标记为完成并归档。审核结论、问题记录、整改结果均沉淀至企业知识库，供后续审核参考。

> **业务价值：** 每一次审核经验都成为可复用的企业资产，而非停留在个人记忆中。

---

### 5.2 审核管理人员的使用流程

```
登录系统 → 审核项目列表查看统计与任务 →（按权限）进入复审或报告中心查阅 → 督办关键问题
```

管理人员与执行人员操作路径一致，不依赖独立驾驶舱。

## 6. 用户期望体验

### 6.1 实际审核人员的期望体验

| 期望 | 描述 |
|------|------|
| **像专业助手，而非管理工具** | 系统应主动提供帮助与建议，而非等待用户逐项填表操作 |
| **减少重复劳动** | 规则匹配、材料整理、报告生成等重复性工作由系统承担 |
| **保留专业判断空间** | AI 给出建议，但最终确认权始终在工程师手中，不强制替代判断 |
| **信息触手可及** | 审核标准、历史案例、证据材料在同一界面关联呈现，无需切换多个工具 |
| **进度清晰可见** | 当前任务处于哪个阶段、还有哪些待办，一目了然 |
| **报告省心省力** | 报告初稿质量足够高，工程师只需修订而非从零撰写 |

### 6.2 审核管理人员的期望体验

| 期望 | 描述 |
|------|------|
| **打开即见全局** | 登录后第一眼看到审核整体态势，无需层层点击 |
| **风险主动呈现** | 高风险问题突出展示，不需要主动搜索才能发现 |
| **数据直观易懂** | 用图表和摘要代替冗长的表格与文字报告 |
| **督办有据可依** | 查看整改进度时，能追溯到具体问题和责任人 |
| **低学习成本** | 无需培训即可理解界面信息，快速上手查阅 |

### 6.3 共同期望

- **专业可信：** 界面与交互体现专业审核场景，而非通用办公工具
- **高效流畅：** 核心操作路径短，无明显等待与卡顿
- **经验可沉淀：** 每一次使用都在积累企业知识，而非仅完成单次任务
- **协作顺畅：** 实际审核人员与审核管理人员在同一平台内信息互通，减少重复沟通
