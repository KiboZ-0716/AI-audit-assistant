## 1. 产品功能架构概述

### 1.1 产品解决的业务问题

在企业制造过程质量审核场景中，审核人员需要依据分散的审核标准，对业务系统数据及线下文档材料进行逐条比对分析，识别不符合项并输出审核结论。当前面临的核心业务问题包括：

- 审核规则分散，执行依赖个人经验，标准不统一
- 人工整理材料、比对标准、编写报告效率低
- 复杂场景下问题识别存在遗漏风险
- 审核经验难以系统化沉淀与复用
- 集团层面缺乏对多工厂审核进展与风险的整体掌握

本产品以 **企业级 AI 智能工作平台** 定位，将 AI 能力嵌入审核全链路，辅助专业人员完成复杂审核任务，而非替代其专业判断。

### 1.2 核心业务链路

产品功能围绕以下核心业务链路组织：

```
确定审核对象 → 配置审核范围与数据源 → AI 审核执行 → 人工确认审核结果 → 指标计算与问题总结 → 报告输出
```

各阶段对应的业务能力：

| 阶段 | 业务能力 | 核心产出 |
|------|----------|----------|
| 确定审核对象 | 审核项目管理 | 审核项目、审核范围（周期/基地/区域/项目/子要素） |
| 配置数据源 | 审核项目管理 | 文件资料、系统数据、移动端补充/修改三类数据来源 |
| AI 审核执行 | AI 审核执行能力 | 结构化审核结果、问题列表、评分建议 |
| 人工确认 | 审核结果管理 | 经确认的问题清单、评分结果 |
| 指标计算与总结 | AI 审核执行能力 + 审核结果管理 | 符合率、工厂级主要问题总结 |
| 报告输出 | 报告中心 | 审核报告、问题总结报告、打分表、BIP 问题管理表 |

### 1.3 AI 在系统中的作用

AI 在本系统中承担 **智能分析与建议** 角色，不替代审核人员的专业判断。具体职责边界如下：

| AI 负责 | 人工负责 |
|---------|----------|
| 审核资料解析与结构化 | 审核项目创建与范围确认 |
| 审核标准自动匹配 | 数据源配置与资料补充 |
| 问题自动识别与描述生成 | AI 结果的审阅、修改与确认 |
| 问题属性分类建议 | 问题等级与评分的最终裁定 |
| 严重度打分建议 | 整改建议的完善与补充 |
| 工厂级问题聚类归纳 | 符合率计算的手动触发 |
| 审核报告初稿生成 | 报告内容的编辑与定稿 |

**核心原则：AI 提供建议，人工最终确认。**

## 2. 用户角色与功能关系

### 2.1 审核管理人员

**角色定位：** 关注整体审核情况、工厂审核进展与问题风险的管理层用户，并拥有全量数据权限（用于查阅、督办与决策支持）。

| 使用功能 | 使用目标 |
|----------|----------|
| 审核驾驶舱 | 快速掌握整体审核情况、工厂审核进展与问题风险分布 |
| 报告中心（查阅） | 查阅各工厂最终审核报告与问题总结，支撑管理决策 |

**关注重点：** 整体审核情况、工厂审核进展、问题风险。

---

### 2.2 实际审核人员

**角色定位：** 完成审核执行与闭环输出的实际执行者，包含审核工程师与现场审核人员（含移动端操作），负责对 AI 结果进行确认与修订。

| 使用功能 | 使用目标 |
|----------|----------|
| 审核项目管理 — 审核项目创建 | 创建审核项目，配置基础信息、审核周期与负责人 |
| 审核项目管理 — 审核范围配置 | 选择审核体系，配置审核项目与子要素 |
| 审核项目管理 — 数据源配置 | 配置文件资料、系统数据接口；并为现场补充/修改提供协同入口 |
| 审核项目管理 — 审核启动 | 检查配置完整性，启动 AI 审核流程，锁定审核范围 |
| 审核结果管理 — 问题查看 | 查看 AI 识别的问题描述、所属子要素、问题来源、严重程度与评分 |
| 审核结果管理 — AI 分析信息 | 查阅审核标准、审核依据、AI 分析过程与评分建议 |
| 审核结果管理 — 人工确认 | 修改问题描述、问题等级、评分与整改建议，新增遗漏问题 |
| 报告中心 | 编辑审核报告初稿，确认问题总结报告与打分表，导出正式报告 |
| 移动端审核能力 | 上传现场资料/图片；补充与编辑问题清单（文本/语音）并协同完成确认 |

**负责事项：** 审核材料分析、问题确认/修订、整改建议完善、报告输出与闭环推进。

## 3. 一级功能架构

一级功能按照业务流程拆解，以 **用户任务驱动** 而非后台菜单驱动，共六大能力域。

---

### 3.1 审核驾驶舱

**能力定位：** 面向审核管理人员，提供审核全局态势的快速感知能力，帮助管理者在不深入审核细节的情况下掌握整体进展与风险。

**解决的业务问题：** 审核数据分散在各工厂、各项目中，管理层难以快速获取全局视图。

**功能能力：**

| 功能 | 能力说明 | 业务价值 |
|------|----------|----------|
| 审核项目统计 | 汇总当前周期内审核项目总数、进行中/已完成/待启动数量 | 掌握审核工作整体负荷 |
| AI 审核状态 | 展示各项目 AI 审核执行进度（待执行/执行中/已完成） | 了解 AI 处理进展，识别阻塞项 |
| 人工复审状态 | 展示各项目人工确认进度（待确认/确认中/已确认） | 掌握审核结论产出进度 |
| 审核趋势分析 | 按时间维度展示审核完成量、问题发现量等趋势 | 识别审核工作规律与变化 |
| 工厂审核情况 | 按工厂维度展示审核进展、符合率、问题分布 | 横向对比各工厂审核成效 |

---

### 3.2 审核项目管理

**能力定位：** 审核全过程的任务管理入口，承载从项目创建到审核启动的配置与组织工作。

**解决的业务问题：** 审核对象涉及周期、基地、区域、项目、子要素等多维信息，数据来源多样，需在启动 AI 审核前完成结构化配置。

#### 3.2.1 审核项目创建

| 能力 | 说明 |
|------|------|
| 创建审核项目 | 新建一次审核任务，作为后续配置与执行的组织单元 |
| 配置基础信息 | 设置项目名称、所属基地、审核周期等基本信息 |
| 设置审核周期 | 明确本次审核的时间范围 |
| 设置负责人 | 指定实际审核负责人 |

#### 3.2.2 审核范围配置

| 能力 | 说明 |
|------|------|
| 选择审核体系 | 确定本次审核适用的标准体系 |
| 配置审核项目 | 选择需要审核的项目维度 |
| 配置子要素 | 选择需要审核的子要素，形成审核对象列表（已评价/未评价） |

> 对应业务流程：手动选择周期 + 基地 + 区域 + 项目 + 子要素，确定审核对象。

#### 3.2.3 数据源配置

支持三类数据来源，可组合使用：

| 数据来源 | 支持方式 | 说明 |
|----------|----------|------|
| 文件资料 | Word、Excel、图片 | 手动上传附件；图片主要指现场拍摄的线下文件照片 |
| 系统数据 | 业务系统数据接口 | 通过 API 自动获取 QMS、IQS 等业务系统数据；区分已完成数字化与未完成数字化 |
| 移动端补充/修改 | 文本/语音 | 在移动端补充与编辑问题清单，作为审核结果确认的输入 |

#### 3.2.4 审核启动

| 能力 | 说明 |
|------|------|
| 检查配置完整性 | 校验审核范围、数据源等必要配置是否完备 |
| 启动 AI 审核流程 | 触发 AI 审核执行，进入数据获取与分析阶段 |
| 锁定审核范围 | 启动后锁定已配置的审核范围，防止执行过程中范围变更 |

---

### 3.3 AI 审核执行能力

**能力定位：** 系统的 AI 核心能力域，在审核启动后自动执行，无需用户逐项操作。这是本产品区别于传统审核工具的关键能力。

**解决的业务问题：** 人工逐条比对标准、翻阅材料、识别问题效率低且易遗漏。

#### 3.3.1 数据理解

| 能力 | 说明 | 对应流程节点 |
|------|------|-------------|
| 文档解析 | 对上传的 Word、Excel 文档进行内容提取 | Document Extract |
| 图片识别 | 对现场拍摄的线下文件照片进行识别 | Document Extract |
| 数据结构化 | 将不同来源的原始数据统一处理为结构化审核输入 | 数据处理 |

#### 3.3.2 审核标准匹配

| 能力 | 说明 | 对应流程节点 |
|------|------|-------------|
| 调用审核标准 | 根据已确定的审核对象，从审核标准知识库检索适用标准 | Knowledge Retrieval |
| 匹配审核规则 | 将审核标准与结构化审核数据进行关联匹配 | Knowledge Retrieval |
| 提供审核依据 | 为后续问题识别提供标准条款作为判断依据 | Knowledge Retrieval |

#### 3.3.3 AI 问题识别

| 能力 | 说明 | 对应流程节点 |
|------|------|-------------|
| 发现不符合项 | 依据审核标准，从审核数据中识别潜在问题 | LLM 节点（审核判断） |
| 生成问题描述 | 对识别出的问题进行归纳，生成结构化问题描述 | LLM 节点（问题生成） |

问题识别后同步执行 **问题属性分类**，依据问题描述归入约 5 类分类结构：

| 分类 | 示例子类 |
|------|----------|
| 问题管理类 | 问题闭环不足、整改跟踪不足 |
| 流程类 | 流程机制不完善、执行不到位 |
| 数据类 | 数据完整性不足、数据应用不足 |
| 专项管理类 | 专项治理不足 |

#### 3.3.4 AI 评分建议

| 能力 | 说明 | 对应流程节点 |
|------|------|-------------|
| 根据标准提供评分建议 | 依据审核标准对子要素级别进行严重度打分建议 | LLM 节点（问题评分） |
| 给出分析依据 | 为每项评分建议提供对应的标准条款与分析理由 | LLM 节点（问题评分） |

> 业务说明：评分规则业务无法细化，可提供案例供 AI 参考。

#### 3.3.5 AI 结果生成

AI 审核执行完成后，输出以下结果供人工确认：

| 输出项 | 说明 |
|--------|------|
| 问题列表 | 包含问题描述、所属子要素、问题来源、问题分类、严重程度、评分建议 |
| 审核建议 | 针对每个问题的整改方向建议 |
| 分析结果 | 结构化审核结果，含 AI 分析过程与审核依据 |

此外，AI 执行阶段还包含 **工厂级问题聚类归纳**（问题总结 LLM 节点），将问题列表合并归纳为 3-5 个主要问题，格式为「问题名称 + 具体表现」，使用审核报告语言输出。

---

### 3.4 审核结果管理

**能力定位：** 实际审核人员对 AI 分析结果进行人工审阅、确认与修订的工作台，是 **人机协同** 的核心交互环节。

**解决的业务问题：** AI 识别结果需要专业人员结合业务实际情况进行确认，确保审核结论的准确性与权威性。

#### 3.4.1 问题查看

展示 AI 生成的完整问题信息：

| 展示项 | 说明 |
|--------|------|
| 问题描述 | AI 生成的问题描述文本 |
| 所属子要素 | 问题关联的审核子要素 |
| 问题来源 | 问题来源的数据或文件（系统数据/上传文件/移动端补充/修改） |
| 严重程度 | AI 建议的严重度等级 |
| 评分 | AI 建议的子要素评分 |

#### 3.4.2 AI 分析信息

展示 AI 做出判断的依据，支撑实际审核人员专业审阅：

| 展示项 | 说明 |
|--------|------|
| 审核标准 | 匹配到的审核标准条款 |
| 审核依据 | AI 引用标准进行判断的具体依据 |
| AI 分析过程 | AI 从数据到结论的分析逻辑 |
| 评分建议 | AI 给出的评分建议及理由 |

#### 3.4.3 人工确认

实际审核人员对 AI 结果拥有最终确认权，支持以下操作：

| 操作 | 说明 |
|------|------|
| 修改问题描述 | 调整 AI 生成的问题表述，使其更准确 |
| 修改问题等级 | 调整严重程度判断 |
| 修改评分 | 调整子要素评分 |
| 修改整改建议 | 完善或重写整改方向建议 |
| 新增问题 | 补充 AI 未识别到的遗漏问题 |

人工确认完成后，可 **手动触发符合率计算**，系统按以下规则计算：

- 区域符合率 =（各条款评分总和 / 条款评价项 × 10）× 100%
- 工厂总体符合率 =（∑各区域条款评分总和 / ∑各区域条款评价项 × 10）× 100%
- 不涉及或未评价的条款标记为 NA，不计入计算

---

### 3.5 报告中心

**能力定位：** 审核结果的正式输出环节，将经人工确认的审核结论转化为可交付的报告文档。

**解决的业务问题：** 审核报告编写耗时，大量精力花在格式整理与内容汇总上。

#### 3.5.1 最终审核报告

| 能力 | 说明 |
|------|------|
| 自动生成 | 基于确认后的问题清单、符合率、工厂级问题总结，由 AI 结合模板生成报告初稿 |
| 人工编辑 | 实际审核人员对报告内容进行修订与完善 |
| 导出 | 支持报告文档导出 |

**报告内容结构：**

| 章节 | 说明 |
|------|------|
| 审核概述 | 本次审核的对象、范围、周期等基本信息 |
| 亮点工作 | 审核中发现的优秀实践或亮点 |
| 主要问题 | 工厂级问题聚类归纳后的 3-5 个主要问题（问题名称 + 具体表现） |
| 下一步要求 | 针对主要问题提出的整改要求与建议 |

#### 3.5.2 问题总结报告

| 输出 | 说明 |
|------|------|
| 问题清单 | 经人工确认后的完整问题列表，含问题描述、分类、严重度、评分 |

关联输出：**BIP 问题管理表**，用于问题跟踪与管理。

#### 3.5.3 打分表

| 输出 | 说明 |
|------|------|
| 审核项目评分结果 | 各子要素的评分结果汇总，含区域符合率与工厂总体符合率 |

---

### 3.6 移动端审核能力

**能力定位：** 支持现场审核人员在移动场景下完成资料采集与问题补充，与 Web 端审核流程协同。

**解决的业务问题：** 现场审核人员需要在审核现场即时上传资料、补充问题，无法依赖 PC 端操作。

| 功能 | 能力说明 |
|------|----------|
| 我的审核任务 | 查看分配给当前用户的审核任务列表与状态 |
| 资料上传 | 上传现场资料（图片、文档等） |
| 问题补充 | 在现场发现新问题时补充至问题清单 |
| 问题编辑 | 对已有问题进行修改，支持文本输入 |
| 图片上传 | 上传现场拍摄的证据图片 |
| 语音输入 | 通过语音方式输入或修改问题描述 |

> 业务流程要求：移动端需兼容，问题清单支持文本/语音方式修改。

## 4. AI 能力架构

AI 能力作为系统的底层智能引擎，独立服务于上层业务功能，分为知识库能力、大模型能力与人机协同三个层次。

### 4.1 知识库能力

为 AI 审核分析提供标准依据与经验参考。

| 知识类型 | 内容 | 作用 |
|----------|------|------|
| 审核标准 | 审核标准条款、评价规则 | 审核标准匹配、问题识别、评分建议的依据 |
| 制度文件 | 企业质量管理制度、规范文件 | 补充审核判断的制度上下文 |
| 历史案例 | 历史审核问题、处置经验 | 问题总结归纳的参考方向，支持历史数据训练总结 |

知识库通过 **Knowledge Retrieval** 节点在审核执行时按需调用，输入审核对象，输出匹配的审核标准。

### 4.2 大模型能力

大模型能力通过多个 LLM 节点分阶段执行，各节点职责明确：

| 能力 | 应用场景 | 对应节点 |
|------|----------|----------|
| 内容理解 | 解析文档、识别图片、结构化数据 | Document Extract |
| 问题分析 | 识别不符合项、生成问题描述、问题分类 | LLM 节点（审核判断、问题生成） |
| 评分建议 | 严重度打分、子要素评分 | LLM 节点（问题评分） |
| 总结归纳 | 工厂级问题聚类，合并为 3-5 个主要问题 | LLM 节点（问题总结） |
| 报告生成 | 基于确认结果生成审核报告初稿 | LLM + 模板 |

**问题总结归纳规则：**

1. 不逐条罗列问题
2. 将逻辑相近的问题合并
3. 输出 3-5 个主要问题
4. 每个问题按「问题名称 + 具体表现」格式输出
5. 使用审核报告语言
6. 归纳方向参考专家经验（可通过历史数据训练）

### 4.3 人机协同

| 环节 | AI 角色 | 人工角色 |
|------|---------|----------|
| 资料分析 | 自动解析、结构化 | 配置数据源、补充资料 |
| 标准匹配 | 自动检索匹配 | 确认审核范围 |
| 问题识别 | 自动识别并生成描述 | 审阅、修改、新增问题 |
| 评分 | 提供评分建议与分析依据 | 确认或调整评分 |
| 问题总结 | 聚类归纳主要问题 | 审阅归纳结果 |
| 符合率计算 | 按公式自动计算 | 手动触发计算 |
| 报告生成 | 生成报告初稿 | 编辑、修订、定稿 |

**原则：AI 在每个环节提供建议与初稿，人工拥有最终确认权。**

## 5. 功能模块关系

### 5.1 完整业务链路

以下为产品功能的完整串联关系，体现从项目创建到报告输出的端到端流程：

```
审核项目创建
    │  配置基础信息、周期、负责人
    ▼
审核范围配置
    │  选择审核体系、项目、子要素（周期/基地/区域/项目/子要素）
    ▼
数据源配置
    │  配置文件资料（Word/Excel/图片）、系统数据（QMS/IQS 等 API）、移动端补充/修改（文本/语音）
    ▼
审核启动
    │  检查配置完整性 → 启动 AI 审核 → 锁定审核范围
    ▼
AI 审核执行
    │  数据理解 → 标准匹配 → 问题识别 → 问题分类 → 评分建议 → 问题总结
    ▼
审核结果确认
    │  实际审核人员审阅 AI 结果 → 修改/确认问题 → 调整评分 → 手动触发符合率计算
    ▼
报告生成
    │  生成审核报告初稿 → 人工编辑 → 输出报告/问题清单/打分表/BIP 问题管理表
```

### 5.2 角色与模块的参与关系

| 流程阶段 | 审核管理人员 | 实际审核人员 |
|----------|:---------------:|:--------------:|
| 审核项目创建 | — | ● |
| 审核范围配置 | — | ● |
| 数据源配置 | — | ●（包含文件资料/系统数据配置与移动端协同补充） |
| 审核启动 | — | ● |
| AI 审核执行 | — | — |
| 审核结果确认 | — | ● |
| 符合率计算 | — | ●（手动触发） |
| 报告生成 | ○（查阅） | ● |
| 审核驾驶舱 | ● | ○ |

> ● 主要参与者　○ 辅助参与者　— 不参与

### 5.3 模块间数据流转

| 上游模块 | 下游模块 | 传递数据 |
|----------|----------|----------|
| 审核项目管理 | AI 审核执行能力 | 审核对象、审核范围、数据源配置 |
| AI 审核执行能力 | 审核结果管理 | 问题列表、评分建议、分析结果、AI 分析依据 |
| 审核结果管理 | 报告中心 | 经确认的问题清单、评分结果、符合率 |
| AI 审核执行能力 | 报告中心 | 工厂级主要问题总结 |
| 审核结果管理 | 审核驾驶舱 | 审核进度、复审状态、符合率指标 |
| 移动端审核能力 | 审核结果管理 | 现场补充的问题、上传的资料 |

### 5.4 能力域依赖关系

```
审核驾驶舱 ←── 依赖 ── 审核项目管理、审核结果管理、报告中心 的状态与结果数据

审核项目管理 ──→ 驱动 ──→ AI 审核执行能力

AI 审核执行能力 ──→ 产出 ──→ 审核结果管理

审核结果管理 ──→ 驱动 ──→ 报告中心

移动端审核能力 ──→ 汇入 ──→ 审核结果管理（问题补充与资料上传）
```
