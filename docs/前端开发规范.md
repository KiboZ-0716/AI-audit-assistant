## 0. 开发边界（必须遵守）

### 0.1 做

- 前端页面与路由
- 页面交互与状态变化
- Mock 数据与模拟业务流程
- AI 执行过程的前端模拟（进度、阶段、结果展示）

### 0.2 不做

- 后端服务、数据库、真实 API
- 真实 AI 模型调用、文件解析服务、数据处理服务
- 真实用户权限 / 登录鉴权体系（可用简单角色切换 Mock）
- 为生产扩展预留的过度架构（网关、微服务适配层等）

### 0.3 原则

> **演示完整性优先，技术完备性其次。**  
> 一切「AI 能力」均为前端状态机 + Mock 结果，对客户表现为真实业务流程。

## 1. 前端技术方案

### 1.1 技术栈

| 层级 | 选型 | 说明 |
|------|------|------|
| 框架 | React | 与项目规则一致 |
| 语言 | TypeScript | 类型约束业务模型与页面状态 |
| 样式 | Tailwind CSS | 快速落实企业级视觉，避免模板 Admin 皮肤依赖 |
| 构建 | Vite | 适合 Demo 快速启动与热更新 |
| 路由 | React Router | 对应独立页面 P1–P5、M1–M2 |
| 图标 / 字体 | 按视觉规范选用克制企业向资源 | 不引入花哨动效库作为默认依赖 |

### 1.2 状态与数据方案（Demo）

| 能力 | 方案方向 | 说明 |
|------|----------|------|
| 全局演示状态 | 轻量客户端状态管理（如 Zustand，或等价轻方案） | 管理当前审核项目、流程阶段、角色切换、重置 Demo |
| 服务端请求 | **不使用** | 禁止接真实后端；不允许假 API 层伪装成待接后端 |
| 数据读取 | Mock 模块同步/短延迟 Promise | 可用短延迟模拟加载，增强演示真实感 |
| AI 执行 | 前端状态机 + 定时推进 | 模拟分阶段进度，禁止真实模型调用 |
| 持久化 | 可选 sessionStorage / localStorage | 仅用于演示中刷新不丢主故事；提供一键重置 |

### 1.3 路由与端形态

| 端 | 路由策略 | 说明 |
|----|----------|------|
| Web | 独立路由对应 P1–P5 | 主演示舞台 |
| 移动端 | 独立路由对应 M1–M2 | 可用窄屏布局或独立路径前缀（如 `/m/*`）演示；不强制真机适配工程化 |

### 1.4 明确禁止的技术方向

- 引入 Ant Design Pro / 各类 Admin 脚手架默认布局作为产品壳
- 为「以后接后端」先写完整 API Service / Request 拦截器体系
- 引入真实上传、OCR、大模型 SDK
- 过度拆分微前端或复杂权限中间件

### 1.5 质量基线（Demo 足够即可）

- TypeScript 严格模式建议开启
- 核心流程可重复演示、可一键重置
- 主路径无阻断性报错
- 不强制完整单测覆盖；关键状态流转建议可手工验收清单化

## 2. 项目目录结构

以下为推荐信息架构，表达「放什么」，不要求文件级一字不差照搬，但职责边界应保持清晰。

```
src/
├── app/                    # 应用入口、路由、全局布局壳
│   ├── routes              # 路由表（P1–P5、M1–M2）
│   └── layouts             # Web 布局 / 移动端子布局（克制，非 Admin 模板）
│
├── pages/                  # 路由级页面（与页面架构一一对应）
│   ├── projects/           # P2 审核项目列表
│   ├── project-workspace/  # P3 审核项目工作台（配置 / AI审核中）
│   ├── result-workspace/   # P4 人工复审工作台
│   ├── report-workspace/   # P5 报告中心（四类产出）
│   └── mobile/             # M1 / M2
│
├── features/               # 按业务能力组织的功能单元（非页面路由）
│   ├── audit-project/      # 项目创建、范围、数据源、启动相关逻辑
│   ├── ai-execution/       # AI 执行状态机、阶段定义、进度推进
│   ├── issue-review/       # 问题确认、修订、依据展示、符合率
│   ├── report/             # 报告内容组装与类型切换
│   └── cockpit/            # 驾驶舱汇总数据映射
│
├── components/             # 可复用 UI 单元（视觉层，避免堆业务）
│   ├── ui/                 # 基础视觉元件（按钮、标签、分割等）
│   └── business/           # 跨页业务展示块（状态徽章、AI建议标识等）
│
├── mocks/                  # 全部演示数据与剧本
│   ├── seed/               # 初始种子数据
│   ├── scenarios/          # 演示剧本（主路径/快捷路径）
│   └── factories/          # 生成器（新建项目时灌入默认 Mock）
│
├── stores/                 # 全局/跨页状态
├── types/                  # 领域类型（项目、问题、报告、阶段枚举等）
├── lib/                    # 工具方法（格式化、延迟、重置）
├── styles/                 # 全局样式与设计 token 映射（若需要）
└── assets/                 # 静态资源
```

### 2.1 目录职责规则

| 目录 | 放 | 不放 |
|------|----|------|
| `pages/` | 页面编排、状态接入、主任务流 | 大量可复用展示细节 |
| `features/` | 某业务能力的状态流转与领域逻辑 | 路由定义 |
| `components/` | 复用展示与轻交互 | 完整业务流程编排 |
| `mocks/` | 所有演示数据与剧本 | 页面 JSX |
| `stores/` | 跨页共享的 Demo 状态 | 真实 API 缓存层 |

### 2.2 与页面架构的映射

| 页面编号 | 目录落点 |
|----------|----------|
| P2 | `pages/projects` |
| P3 | `pages/project-workspace` |
| P4 | `pages/result-workspace` |
| P5 | `pages/report-workspace` |
| M1 / M2 | `pages/mobile` |

> 已取消 P1 `dashboard` 作为正式模块；历史目录若存在仅作废弃，不再扩展。

## 3. 页面开发顺序

开发顺序以 **客户演示主路径可跑通** 为第一目标，而不是按菜单完整性铺开。

### 3.1 阶段划分

#### 阶段 A：骨架与主故事线（必须先完成）

1. 应用壳与路由（Web：无驾驶舱）  
2. 领域类型 + Mock 种子 + 全局 Store  
3. **P2 审核项目列表**（创建、统计、四态进入）  
4. **P3 审核项目工作台**（左树右配、启动锁定、AI 三步、直达 P4）  
5. **P4 人工复审**（改描述与打分）  
6. **P5 报告中心**（四类产出）  

**阶段 A 完成标准：**  
`P2 → P3 → P4 → P5` 可演示，并可重置。

#### 阶段 B：体验增强

7. 列表图表完善、导航与返回体验  
8. 演示友好细节  

#### 阶段 C：移动端

9. **M1 / M2**（人工录入类现场取证）  

### 3.2 不建议的开发顺序

- 先做全量通用组件库再做页面  
- 先做驾驶舱大屏再做主链路  
- 先做移动端再做 Web 主路径  
- 并行拆太多页导致主路径长期不通  

### 3.3 每页开工前检查

对照文档确认：

- 页面架构中的职责边界  
- 交互方案中的状态与跳转  
- 视觉规范中的气质与禁止项

## 4. Mock 数据组织方式

### 4.1 组织原则

| 原则 | 说明 |
|------|------|
| 单一故事主线 | 默认围绕同一条制造过程质量审核故事，避免多套互斥数据 |
| 数据集中 | 全部 Mock 放在 `mocks/`，页面禁止散落硬编码长文本（少量 UI 文案除外） |
| 可变可重置 | 演示中的修改写在 Store；重置时回到 seed |
| 假装真实 | 文案像审核语言；结构贴近业务对象，但不为后端表结构建模 |

### 4.2 建议的 Mock 分层

| 层级 | 内容 | 用途 |
|------|------|------|
| Seed（种子） | 初始项目列表、预置「待确认/已完成」样例项目、驾驶舱汇总 | 开场与快捷演示 |
| Scenario（剧本） | 主路径新建后的默认范围、数据源、AI 阶段文案、问题清单、报告正文 | 完整走读 |
| Runtime（运行时） | 用户在 Demo 中修改后的问题确认态、符合率、报告切换态 | 交互结果 |
| Factory（工厂） | 「新建审核」时生成新项目 ID 与默认配置 | 可重复演示 |

### 4.3 核心 Mock 对象（逻辑模型）

不规定字段实现，但建议覆盖以下业务对象：

- 审核项目（状态：配置中 / 执行中 / 待确认 / 已完成）  
- 审核范围（周期、基地、区域、项目、子要素）  
- 数据源配置（文件、系统数据、移动端协同标记）  
- AI 执行阶段列表与当前进度  
- 问题清单（描述、分类、严重度、评分建议、来源、确认态、AI 依据）  
- 符合率结果  
- 报告资产（最终报告四段式、问题总结、打分表、BIP 表）  
- 驾驶舱汇总指标  

### 4.4 AI 能力的 Mock 方式

| 真实能力 | Demo 替代 |
|----------|-----------|
| 文档解析 / 图片识别 | 配置完成即视为已具备资料；执行阶段播放「解析中」文案 |
| 标准匹配 / 问题识别 | 定时推进阶段；结束后灌入预置问题列表 |
| 评分与总结 | 预置评分建议与 3–5 条主要问题 |
| 报告生成 | 基于当前确认结果映射到预置报告模板文本（可随修订微调展示） |

**禁止：** 调用任何外部 AI API；用随机胡言生成不可控内容影响演示稳定性。

### 4.5 延迟与节奏

- AI 执行阶段建议可控总时长（便于讲解停顿）  
- 列表切换、符合率计算可用短延迟  
- 提供「演示加速」开关（可选），但不作为默认隐藏主过程  

### 4.6 一键重置

必须提供重置能力，恢复到 Seed / 主剧本初始态，保证多场客户演示可重复。

## 5. 组件设计原则

### 5.1 分层

| 层级 | 职责 | 示例方向 |
|------|------|----------|
| 基础 UI | 无业务语义的视觉元件 | 按钮、标签、分区标题 |
| 业务展示块 | 有审核语义但可复用 | AI 建议标识、项目状态、问题确认态、阶段进度条 |
| 页面编排 | 组合 features + 组件完成任务流 | P3/P4/P5 页面本身 |

### 5.2 设计原则

1. **业务优先于组件库完整度**  
   先保证主路径页面可用，再抽取重复块；禁止先造庞大 UI Kit。

2. **状态可见、语义明确**  
   AI 建议 / 待确认 / 已确认 / 已修订等必须可区分，服务演示讲解。

3. **避免 Admin 模板组件堆砌**  
   不默认用大表格管理后台套路承载所有页面；工作台类页面以任务流组织。

4. **AI 表达克制**  
   AI 相关组件用于标识建议与过程，不做成炫技装饰中心。

5. **复用阈值决定抽取**  
   仅出现一次的结构可留在页面内；跨 P3/P4/P5 复用的状态与标识再下沉到 `components/business`。

6. **组件不拥有跨页真相**  
   跨页数据以 Store + mocks 为准；组件以展示与局部交互为主。

### 5.3 不鼓励

- 为每个业务名词都建「万能 Form 组件」  
- 深层万能配置化页面生成器  
- 复制粘贴第三方 Dashboard 模板组件直接当产品界面

## 6. 页面实现规范

### 6.1 通用规范

| 规范 | 说明 |
|------|------|
| 一页一主任务 | 对齐页面架构：P3 配置与执行、P4 确认、P5 报告 |
| 同页多状态 | P3 用页面状态切换，不拆成多个路由页 |
| 跳转克制 | 按交互方案连通；禁止额外造「管理子页」打断演示 |
| 文案专业 | 使用审核业务语言；突出「AI 建议，人工确认后生效」 |
| 视觉遵从 | 执行 `UI视觉设计规范`：专业、可靠、工业化、克制智能 |

### 6.2 分页面实现要点

#### P2 项目列表

- 展示项目状态，支持新建进入 P3  
- 按状态进入 P3 / P4 / P5  
- 服务于演示入口，不做复杂筛选系统  

#### P3 项目工作台

- 必须实现三态：配置中 / 执行中 / 已完成  
- 配置中：范围 + 数据源同页完成；启动前做完整性提示  
- 执行中：分阶段进度可见，可讲解  
- 启动后范围锁定  
- 完成后提供进入 P4 的主动作  

#### P4 结果工作台

- 问题列表 + 同页依据详情  
- 支持确认 / 修订（Demo 至少可改变确认态）  
- 手动触发符合率并展示结果  
- 明确 AI 建议与已确认的区别  
- 进入 P5  

#### P5 报告工作台

- 最终报告四段结构可读  
- 同页切换其他报告资产  
- 内容与当前确认结果保持演示一致性  

#### P1 驾驶舱

- 基于同一 Store 汇总，避免另起一套无关假数据导致故事分裂  
- 不承担配置与确认主操作  

#### M1 / M2

- 以汇入 P4 为目的；上传与语音可用前端模拟成功反馈  
- 不在移动端实现完整配置/报告闭环  

### 6.3 交互与反馈规范

- 主操作必须有明确反馈（启动、确认、计算、生成/查看报告）  
- AI 执行避免「一秒出结果」；保留可感知过程  
- 错误仅保留演示需要的轻量校验（如未配置完整不能启动）  
- 不实现真实权限拦截；如需角色差异，用显式切换说明  

### 6.4 验收标准（开发完成定义）

主路径验收：

1. 可从 P2 新建并配置 P3  
2. 可启动并看到 AI 分阶段执行  
3. 可进入 P4 查看依据并确认/修订  
4. 可计算符合率  
5. 可进入 P5 查看报告并切换类型  
6. 可一键重置后完整重演  

增强验收（阶段 B/C）：

7. P1 能反映项目进展叙事  
8. M2 补充后能在 P4 体现来源  

视觉验收：

9. 整体不呈现普通 Admin 模板感  
10. AI 与人工确认关系可被讲解指认

## 7. 协作与文档对齐

| 变更类型 | 处理方式 |
|----------|----------|
| 增删独立页面 | 先改页面架构与交互方案，再改路由与目录 |
| 调整演示剧本 | 优先改 `mocks/`，避免页面内分叉数据 |
| 视觉风格调整 | 对齐 UI 视觉设计规范后统一改 token/基础样式 |
| 想接真实后端 | **本阶段拒绝**；单独立项，不在本 Demo 规范内扩展 |
